# Provenance Guard

A backend system that classifies submitted text as likely AI-generated,
likely human-written, or uncertain — with a confidence score, a
plain-language transparency label, an appeals workflow, rate limiting, and a
structured audit log. Built for AI201 Project 4.

Full design rationale lives in [planning.md](planning.md). This README is
the evidence record — the exact label text, real test scores, and rate-limit
behavior graders need.

## Setup

```bash
python -m venv .venv
source .venv/Scripts/activate     # Windows Git Bash
# or: source .venv/bin/activate   # Mac/Linux
pip install -r requirements.txt
cp .env.example .env              # then fill in GROQ_API_KEY
python app.py
```

Server runs at `http://localhost:5000`.

The response always includes `attribution`, `confidence`, and `label` — the three fields
required to both communicate the verdict and give the creator the information needed to
decide whether to appeal. `signals` shows both individual scores alongside the combined
`confidence` so the reasoning is auditable.

## Architecture Overview

A submission's path from input to label:

1. `POST /submit` receives `{text, creator_id}`, generates a `content_id`.
2. The text is run through **both** detection signals: a Groq LLM judgment
   call and a pure-Python stylometric analysis (see below).
3. The two scores are blended into one `combined_score` (the `confidence`
   field), which is mapped to one of three `attribution` buckets via
   threshold rules.
4. A `label` string is rendered from the bucket + score.
5. The submission and a matching audit-log row are written to SQLite.
6. The JSON response (`content_id`, `attribution`, `confidence`, `label`,
   both signal scores) is returned.

A creator who disagrees can `POST /appeal` with the `content_id` + their
reasoning; that flips `status` to `under_review` and writes a second
audit-log row referencing the same `content_id`, so the original decision
and the contest of it sit side by side. See `## Architecture` in
[planning.md](planning.md) for the full ASCII diagram of both flows.

## Detection Signals

| Signal | What it measures | Why chosen | Blind spot |
|---|---|---|---|
| **LLM judgment** (Groq `llama-3.3-70b-versatile`) | Holistic semantic/stylistic coherence — generic hedging, overly balanced framing, formulaic transitions, lack of personal point of view | Strongest single signal in practice; catches *meaning-level* tells no statistical method can see | Black box, can be inconsistent, trivially defeated by prompting a generator to "write casually" |
| **Stylometric heuristics** (pure Python) | Sentence-length uniformity (coefficient of variation), lexical diversity (type-token ratio), AI marker-phrase density | Deterministic, explainable, free, and structural — a genuinely different property than the LLM's semantic judgment | Says nothing about meaning; a terse formal human writer (technical writing, legal prose, non-native English) can score AI-like on uniformity alone |

These are combined as `0.65 * llm_score + 0.35 * stylometric_score`. Full
metric definitions are in [planning.md](planning.md) section 1.

## Confidence Scoring

`confidence` is a single float in `[0, 1]`, meaning "estimated probability
this is AI-generated." Thresholds are **deliberately asymmetric**: a false
accusation (calling a human's work AI) does real reputational harm, so the
bar to declare `likely_ai` (`>= 0.70`) is higher than the bar to declare
`likely_human` (`<= 0.40`), leaving a wide 30-point `uncertain` band in
between that the system falls into by default when evidence is mixed.

**How I validated the scores are meaningful:** I ran four hand-picked test
passages spanning the confidence range (the same ones used during
Milestone 4) and confirmed the combined score landed where intuition says
it should — and, importantly, that a realistic *false-positive trap* (formal
academic writing) landed in `uncertain` rather than `likely_ai`, which is
the asymmetric design working as intended:

| Input | llm_score | stylometric_score | **confidence** | attribution |
|---|---|---|---|---|
| "In today's fast-paced world, it is important to note..." (AI cliché-loaded) | 0.90 | 0.4158 | **0.7305** | `likely_ai` |
| "ok so i finally tried that new ramen place..." (casual human) | 0.20 | 0.0 | **0.13** | `likely_human` |
| Formal academic econ paragraph (real human writing, false-positive trap) | 0.80 | 0.1787 | **0.5825** | `uncertain` |
| Lightly-edited AI text about remote work | 0.70 | 0.0991 | **0.4897** | `uncertain` |

Two examples worth calling out for the "high-confidence vs. lower-confidence
produce different labels" requirement:
- **High-confidence case:** the AI-cliché passage scores **0.73** →
  `likely_ai` → reader sees the AI-generated label.
- **Lower-confidence case:** the formal academic paragraph scores **0.58**
  → `uncertain` → reader sees the uncertainty label, *not* an AI accusation,
  even though the raw LLM score alone (0.80) would have crossed an
  AI threshold on its own. The stylometric signal pulled it back down —
  this is the multi-signal blend doing its job.

## Transparency Label

Exact text shown to a reader (`{pct}` is the live percentage computed from
`confidence`):

| Attribution | Exact label text |
|---|---|
| `likely_ai` (high-confidence AI) | `🤖 Likely AI-Generated — Our system estimates a {pct}% likelihood that this content was generated by AI, based on language-model analysis and writing-style patterns. If you wrote this yourself, you can appeal this classification.` |
| `likely_human` (high-confidence human) | `✍️ Likely Human-Written — Our system estimates a {pct}% likelihood that this content was written by a human, based on language-model analysis and writing-style patterns. No action is needed.` |
| `uncertain` | `❓ Uncertain Origin — Our system could not confidently determine whether this content was written by a human or AI (AI-likelihood: {pct}%). This reflects genuine uncertainty, not suspicion of wrongdoing.` |

Real examples from the table above:
- `likely_ai` at confidence 0.73 → *"🤖 Likely AI-Generated — Our system estimates a 73% likelihood that this content was generated by AI, based on language-model analysis and writing-style patterns. If you wrote this yourself, you can appeal this classification."*
- `likely_human` at confidence 0.13 → *"✍️ Likely Human-Written — Our system estimates a 87% likelihood that this content was written by a human, based on language-model analysis and writing-style patterns. No action is needed."* (note: 87% = 1 − 0.13, the label reports confidence in *its own* direction, not raw AI-probability)
- `uncertain` at confidence 0.58 → *"❓ Uncertain Origin — Our system could not confidently determine whether this content was written by a human or AI (AI-likelihood: 58%). This reflects genuine uncertainty, not suspicion of wrongdoing."*

## Appeals Workflow

`POST /appeal` with `{content_id, creator_reasoning}`:

```bash
curl -s -X POST http://localhost:5000/appeal \
  -H "Content-Type: application/json" \
  -d '{"content_id": "ce21353c-4723-4ebb-9b30-e51bbbfda4ad", "creator_reasoning": "This is my own academic writing from a paper I published. I write in a formal register because that is the convention in economics, not because I used AI."}'
```
```json
{
  "content_id": "ce21353c-4723-4ebb-9b30-e51bbbfda4ad",
  "status": "under_review",
  "timestamp": "2026-06-29T03:34:31.188425+00:00",
  "message": "Appeal received and logged. Content status set to under_review."
}
```

This updates the submission's `status` to `under_review` and writes a new
audit-log row (`event_type: "appeal"`) carrying the original
`attribution`/`confidence`/signal scores alongside the new
`appeal_reasoning` — see the audit log sample below, entry `id: 5`.
Automated re-classification is out of scope; this just gets the contested
decision in front of a human reviewer with full context.

## Rate Limiting

`/submit` is limited to **8 requests/minute and 60 requests/day** per IP
(Flask-Limiter, in-memory storage).

**Reasoning:** a real writer submitting and re-submitting edited drafts in
one sitting rarely exceeds a handful of requests per minute — 8/minute
gives comfortable headroom for that without allowing a script to flood the
endpoint. 60/day covers a prolific creator submitting many short pieces
across a workday while bounding both Groq API cost and SQLite growth from a
single source.

**Evidence — 12 rapid requests against the limit:**
```

```
First 8 succeed, remainder are rejected with `429 Too Many Requests`.

## Known Limitations

**Formal/technical human writing risks an over-confident "uncertain" rather
than a clean "likely_human."** The stylometric uniformity metric penalizes
low sentence-length variance, which is a structural feature of formal
registers (academic writing, legal prose, non-native-English phrasing), not
necessarily evidence of AI authorship. In testing, a real human-written
economics paragraph scored 0.58 (`uncertain`) rather than a confident
`likely_human` — the asymmetric thresholds prevented a false *accusation*,
but the system still couldn't confidently *exonerate* it either. A
deployed version of this system would want a separate "formal register"
detector to avoid systematically under-confident scores for technical
writers.

**Very short submissions** (a haiku, a 2-sentence flash-fiction fragment)
make the stylometric signal nearly meaningless — sentence-length variance
and type-token ratio need several sentences to be statistically stable, so
with 1-2 sentences the stylometric score degrades toward a neutral ~0.5,
and the system ends up leaning almost entirely on the LLM signal (which is
itself less reliable on very short text). This isn't a generic "needs more
data" complaint — it's a direct consequence of the two specific metrics
chosen requiring multiple sentences to mean anything.

## Spec Reflection

**How the spec helped:** Writing out the exact label text and threshold
table in planning.md *before* writing scoring.py forced a concrete decision
("0.51 must look meaningfully different from 0.95") that would have been
easy to fudge if I'd started from code. It's also what surfaced the
human-label percentage bug early — the spec said "confidence the reader
should find meaningful," and checking the rendered string against that
spec immediately showed `likely_human` displaying "13% likely human" for a
0.13 AI-probability score, which is backwards.

**Where implementation diverged from the spec:** The original plan used
`0.6/0.4` signal weights and `0.80/0.35` attribution thresholds. When I ran
the four Milestone-4 test passages against that spec, the deliberately
AI-cliché-loaded sample only reached a combined score of ~0.64 — below the
0.80 bar — so `likely_ai` was effectively unreachable by any realistic
input, because the stylometric uniformity sub-metric caps out well below
1.0 for naturally varied prose even when stuffed with AI marker phrases. I
recalibrated to `0.65/0.35` weights and `0.70/0.40` thresholds (documented
in planning.md), re-ran all four test cases, and confirmed all three label
buckets were reachable while keeping the asymmetric-threshold protection
against false AI accusations intact.

## AI Usage

1. **Threshold recalibration (see Spec Reflection above).** I directed the
   AI tool to implement `scoring.py` exactly per the original planning.md
   spec (`0.6/0.4` weights, `0.80/0.35` thresholds). It implemented the spec
   correctly, but when we ran it against the four Milestone-4 test passages
   the obviously-AI-cliché sample only reached ~0.64 — never crossing the
   0.80 bar, making `likely_ai` practically unreachable. I had it
   recalibrate to `0.65/0.35` weights and `0.70/0.40` thresholds and re-run
   all four test cases to confirm every label bucket was actually reachable
   before moving on.
2. **`render_label()` percentage bug.** I directed the AI tool to implement
   `labels.py` from the three label templates in planning.md. The first
   version computed the displayed `{pct}` the same way for all three
   templates (`combined_score * 100`). Testing it directly against sample
   scores (`classify(0.13)` → `likely_human`) surfaced that this produced
   "Our system estimates a 13% likelihood that this content was written by
   a human" for a passage that was 87% likely human — backwards. I had it
   fix `render_label()` to compute `pct = (1 - combined_score) * 100`
   specifically for the `likely_human` branch before wiring it into the
   `/submit` route.
